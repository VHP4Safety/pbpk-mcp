# PBPK MCP v0.3.1

## Release Summary

- `PBPK MCP v0.3.1: executable qualification evidence and generic evidence bundles`

## Overview

PBPK MCP `v0.3.1` strengthens the OECD-oriented qualification layer without changing the public MCP workflow.

This increment focuses on three things:

- executable verification evidence that can be run and stored per loaded simulation
- conservative classification of performance evidence so runtime smoke is not overstated as predictive validation
- generic companion evidence bundles for performance and uncertainty, so researchers can attach structured qualification evidence to `.pkml` or MCP-ready `.R` models without editing bridge code

## Added

- `run_verification_checks` for executable verification across loaded models
- stored `executableVerification` snapshots in `export_oecd_report`
- generic companion performance bundles such as `model.performance.json`
- generic companion uncertainty bundles such as `model.uncertainty.json`
- reusable public templates:
  - `examples/performance_evidence_bundle.template.json`
  - `examples/uncertainty_evidence_bundle.template.json`
- conservative semantic validation for bundled performance and uncertainty evidence rows

## Changed

- OECD report export now separates:
  - runtime capability and model profile
  - static verification evidence and platform qualification evidence
  - stored executable verification evidence
  - internal/runtime evidence vs predictive or externally qualified performance evidence
- static manifest inspection now treats companion performance and uncertainty bundles as valid supplemental evidence for supported models
- release-readiness and live-stack checks now exercise executable verification and richer evidence export paths

## Why It Matters

This release keeps PBPK MCP generic.

The new evidence-bundle path is intentionally model-agnostic:

- researchers can attach structured performance or uncertainty evidence to their own PBPK models
- the MCP surfaces those bundles conservatively through static manifest validation and OECD report export
- the MCP does not silently upgrade incomplete rows into stronger scientific claims

That improves traceability and portability without making the platform dependent on one example model or one backend.

## Qualification Boundary

`v0.3.1` improves qualification structure and evidence plumbing, but it does not make every model scientifically qualified by default.

Important limits remain:

- executable verification is not a substitute for a full implementation-verification dossier
- bundled evidence can still be internal or runtime-only evidence rather than observed-versus-predicted validation
- uncertainty bundles do not imply full uncertainty quantification or externally reviewed qualification studies
- `.pksim5` and `.mmd` remain conversion sources, not direct runtime formats

## Verification

The release-prep validation path for this increment should include:

```bash
python3 -m unittest -v tests/test_model_manifest.py tests/test_oecd_bridge.py
python3 -m unittest -v tests/test_oecd_live_stack.py
python3 scripts/release_readiness_check.py --skip-unit-tests
```

For a release build, also run the Docker-backed model smoke workflow and retain the uploaded smoke artifact with the release evidence.
