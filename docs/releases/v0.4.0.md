# PBPK MCP v0.4.0

## Release Summary

- `PBPK MCP v0.4.0: formalized contract artifacts and major patch-debt reduction`

## Overview

PBPK MCP `v0.4.0` turns the public `pbpk-mcp.v1` contract into a proper published artifact set and moves the local runtime materially closer to a packaged implementation.

This release adds:

- published PBPK-side object schemas, examples, a capability matrix, and a machine-readable contract manifest
- integrity-hashed live MCP resources for the published contract artifacts
- hardened packaging and release gates across installed-package, `sdist`, and wheel boundaries
- a hardened local deployment overlay with explicit auth requirements
- major `0.4.x` migration of the generic MCP runtime surface into packaged `src/`

It also:

- retires the tracked patch-copy flow for the generic Python contract surface
- keeps the public PBPK workflow and PBPK/NGRA boundary intact
- keeps BER calculation and assessment decisions outside PBPK MCP

It does **not**:

- add new PBPK decision policy
- publish local/example reference_compound assets that remain intentionally private
- claim a fully packaged default runtime boundary for local development yet

## Added

- versioned JSON Schemas and example payloads for the PBPK-side NGRA object family
- a dedicated capability matrix and machine-readable contract manifest
- `/mcp/resources/schemas`, `/mcp/resources/capability-matrix`, and `/mcp/resources/contract-manifest` with integrity hashes
- hardened deployment profile support through `docker-compose.hardened.yml` and `scripts/deploy_hardened_stack.sh`
- packaging and release gates for dependency preflight, installed-package validation, distribution validation, release-metadata validation, and retained release-artifact evidence

## Changed

- generic route/resource logic, tool-registry logic, workflow tools, and the generic adapter/runtime boundary now live in packaged `src/`
- packaged `mcp_bridge.contract` artifacts are now the authoritative runtime source for the published contract resources
- the worker image now bakes baseline runtime assets directly, and the local stack overlays the workspace through bind mounts plus `scripts/runtime_src_overlay.pth`
- deployment and architecture docs now describe the real source-overlay development/runtime model instead of the retired tracked patch implementation layer

## Removed

- the tracked patch implementation layer for generic Python contract modules
- the last live patch-copy/install step from the local deployment flow
- the dependency on copied generic contract JSON under `/app/var/contract` for live contract resources

## Why It Matters

PBPK MCP is now materially easier to validate, package, and adopt:

- downstream tooling can consume a real published contract instead of inferring it from code and prose
- maintainers can prove that source checkouts, installed packages, distributions, and live MCP resources all agree on the same artifact inventory
- the local runtime path is significantly simpler and closer to the eventual packaged boundary
- the public PBPK/NGRA split remains explicit while the platform surface becomes more stable

## Verification

The release was verified with:

```bash
make runtime-patch-check
make runtime-contract-test
python3 -m unittest -v tests/test_model_discovery_live_stack.py tests/test_oecd_live_stack.py
python3 scripts/release_readiness_check.py --skip-unit-tests
```

It was also checked live after redeploy by confirming:

```bash
python3 scripts/wait_for_runtime_ready.py --timeout-seconds 20 --stable-successes 2
curl -s http://127.0.0.1:8000/health | jq .
curl -s http://127.0.0.1:8000/mcp/list_tools | jq '.tools | length'
curl -s http://127.0.0.1:8000/mcp/resources/contract-manifest | jq .
```

Those checks confirmed the live stack served the updated contract-manifest hash for the packaged-contract-authoritative runtime path.
