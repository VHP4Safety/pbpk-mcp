# PBPK MCP v0.4.2

## Release Summary

- `PBPK MCP v0.4.2: worker-image rebuild hotfix for the packaged-default runtime`

## Overview

PBPK MCP `v0.4.2` is a narrow hotfix on top of `v0.4.1`.

It fixes the packaged worker-image rebuild path so fresh local image builds validate the installed package version dynamically from `pyproject.toml`, instead of asserting against a stale hardcoded release string.

This release does **not** change:

- the published `pbpk-mcp.v1` contract
- the public MCP tool catalog
- the PBPK/NGRA boundary
- the packaged-default local runtime shape introduced in `v0.4.1`

## Fixed

- the `rxode2` worker image build now derives the expected installed package version dynamically from `pyproject.toml`
- the deployment-profile regression now enforces that the worker-image version check stays dynamic across future release bumps

## Why It Matters

`v0.4.1` made the local runtime packaged by default, which was the right architectural move, but a fresh worker-image rebuild still failed because the build-time version assertion was left pinned to `0.4.0`.

`v0.4.2` closes that gap. Fresh local image builds, package metadata, and the running packaged stack now agree again.

## Verification

The release was verified with:

```bash
make runtime-patch-check
make runtime-contract-test
python3 -m unittest -v tests/test_deployment_profiles.py
./scripts/build_rxode2_worker_image.sh
python3 scripts/release_readiness_check.py --skip-unit-tests
python3 -m unittest -v tests/test_model_discovery_live_stack.py tests/test_oecd_live_stack.py
```

It was also checked live after rebuilding and recreating the local stack by confirming:

```bash
curl -s http://127.0.0.1:8000/health | jq .
curl -s http://127.0.0.1:8000/mcp/resources/contract-manifest | jq .
```

Those checks confirmed the live packaged stack rebuilt and served `version: 0.4.2` successfully.
